Advanced Golang Questions

Internals, concurrency patterns, interfaces, and runtime behavior

01 Explain how Go's goroutine scheduler (GMP model) works under the hood. Hard

Go uses a GMP model: G (Goroutine), M (OS Thread), P (Processor/Logical CPU). Each P has a local run queue of goroutines. Ms execute Gs via Ps.

  • G — lightweight goroutine (~2KB initial stack, grows dynamically)
  • M — OS thread; can block on syscalls without blocking others
  • P — context; holds local run queue and caches; count = GOMAXPROCS

When a G blocks on a syscall, the P detaches from M and finds/creates another M to continue running other Gs (handoff). Work-stealing: idle Ps steal goroutines from other Ps' run queues.

go
// Set the number of OS threads that can execute simultaneously
import "runtime"

func main() {
    runtime.GOMAXPROCS(4) // 4 Ps = 4 goroutines truly parallel
    
    // Check number of current goroutines
    n := runtime.NumGoroutine()
    fmt.Println("goroutines:", n)
}
💡 Key insight: GOMAXPROCS defaults to the number of CPU cores since Go 1.5. Increasing it beyond CPU count rarely helps and adds scheduling overhead.
02 What is the difference between a goroutine leak and how do you detect/prevent it? Hard

A goroutine leak occurs when goroutines are spawned but never terminate — they stay blocked on channels, mutexes, or waiting for I/O that never completes, consuming memory and CPU.

go — LEAK example
// ❌ Goroutine leaks: channel send with no receiver
func leaky(ch chan int) {
    go func() {
        ch <- 42 // blocks forever if no one reads
    }()
}

// ✅ Fix: use context for cancellation
func safe(ctx context.Context, ch chan int) {
    go func() {
        select {
        case ch <- 42:
        case <-ctx.Done():
            return // goroutine exits cleanly
        }
    }()
}

Detection:

  • Use runtime.NumGoroutine() in tests to check before/after
  • Use goleak library (Uber) in unit tests
  • Use pprof goroutine profile endpoint in production
⚠ Common causes: Forgotten goroutines blocked on unbuffered channels, http handlers without timeouts, range over unclosed channels.
03 How do interfaces work internally in Go? What are the performance implications of interface dispatch? Medium

An interface value in Go is a two-word data structure: itab pointer (type info + method table) and a data pointer (to the concrete value or a heap allocation).

go
// Internal representation (simplified)
type iface struct {
    tab  *itab         // points to type info + method table
    data unsafe.Pointer // points to concrete data
}

// Small values stored inline (via pointer to stack copy)
// Large values heap-allocated on interface assignment

type Animal interface { Speak() string }

type Dog struct{}
func (d Dog) Speak() string { return "woof" }

var a Animal = Dog{}  // itab cached after first use

Performance implications:

  • Interface method call = indirect function call through vtable (~2-3x slower than direct call)
  • Assigning concrete type to interface can cause heap escape → GC pressure
  • itab is cached per (interface, concrete type) pair — not reconstructed every call
  • Type assertions have O(1) cost; type switches have O(n) unless reordered by compiler
💡 Tip: Use concrete types in hot paths. Reserve interfaces for API boundaries. Benchmark with go test -bench -gcflags="-m" to see escape analysis.
04 Explain the difference between select with default, select without default, and time.After. What are the pitfalls? Medium
go
// 1. select without default — BLOCKS until a case is ready
select {
case v := <-ch:
    fmt.Println(v)
}

// 2. select with default — NON-BLOCKING, falls to default if no case ready
select {
case v := <-ch:
    fmt.Println(v)
default:
    fmt.Println("no message")
}

// 3. time.After — creates a channel, fires after duration
// ❌ PITFALL: leaks timer until it fires even after select completes
select {
case v := <-ch:
    fmt.Println(v)
case <-time.After(1 * time.Second): // timer goroutine leaked until 1s
    fmt.Println("timeout")
}

// ✅ FIX: reuse timer with Reset()
timer := time.NewTimer(1 * time.Second)
defer timer.Stop()
select {
case v := <-ch:
    fmt.Println(v)
case <-timer.C:
    fmt.Println("timeout")
}
⚠ Pitfall: time.After in a tight loop causes significant GC pressure — each call allocates a new timer that's only GC'd after it fires.
05 What are Go generics (type parameters)? How do they differ from interface-based polymorphism? Medium

Introduced in Go 1.18, generics allow type-parameterized functions and types resolved at compile time, eliminating runtime overhead of interface dispatch for type-agnostic algorithms.

go
// Generic function with constraint
func Map[T, U any](s []T, f func(T) U) []U {
    result := make([]U, len(s))
    for i, v := range s {
        result[i] = f(v)
    }
    return result
}

// Type constraint using union
type Number interface {
    ~int | ~float64 | ~int64
}

func Sum[T Number](nums []T) T {
    var total T
    for _, n := range nums {
        total += n
    }
    return total
}

// Usage — no boxing/unboxing overhead
ints := []int{1, 2, 3}
fmt.Println(Sum(ints)) // 6

Generics vs. Interfaces:

  • Generics: compile-time monomorphization (or GC shapes), zero boxing overhead
  • Interfaces: runtime dispatch, heap escape possible, more flexible for heterogeneous collections
  • Use generics for algorithms; use interfaces for behavior abstraction
06 Explain defer execution order, defer with named return values, and the panic/recover mechanism. Hard
go
// Defer is LIFO (last in, first out)
func example() {
    defer fmt.Println("1")
    defer fmt.Println("2")
    defer fmt.Println("3")
    // prints: 3, 2, 1
}

// Named return — defer can MODIFY return value
func double(x int) (result int) {
    defer func() {
        result *= 2 // modifies named return!
    }()
    result = x
    return // returns x*2, not x
}

// panic / recover — must be in same goroutine
func safeDiv(a, b int) (result int, err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("recovered: %v", r)
        }
    }()
    return a / b, nil // panics if b == 0
}
⚠ Gotcha: Defer evaluates arguments immediately but runs the function later. defer fmt.Println(i) captures the current value of i, not its final value.
07 What is the difference between sync.Mutex, sync.RWMutex, and sync.Map? When to use each? Medium
go
// sync.Mutex — exclusive lock, all ops serialized
var mu sync.Mutex
mu.Lock()
// critical section
mu.Unlock()

// sync.RWMutex — multiple readers OR one writer
var rw sync.RWMutex

// Read lock (shared)
rw.RLock()
val := cache[key] // concurrent reads OK
rw.RUnlock()

// Write lock (exclusive)
rw.Lock()
cache[key] = val
rw.Unlock()

// sync.Map — specialized for high-concurrency, mostly-read workloads
var m sync.Map
m.Store("key", "value")
v, ok := m.Load("key")
m.LoadOrStore("key", "default")
m.Range(func(k, v any) bool {
    fmt.Println(k, v)
    return true
})

When to use each:

  • Mutex: Write-heavy workloads, simple cases, when reads and writes are balanced
  • RWMutex: Read-heavy workloads (10:1+ read/write ratio), caches
  • sync.Map: Keys written once then read many times, or many disjoint goroutines with non-overlapping keys
💡 Benchmark note: sync.Map has higher overhead per-op than a plain map+mutex for low contention. Always benchmark before switching.
08 Explain channels as first-class values, channel directions, and the fan-out/fan-in concurrency patterns. Medium
go
// Channel directions restrict usage at compile time
func producer(out chan<- int) { // send-only
    for i := 0; i < 5; i++ { out <- i }
    close(out)
}

func consumer(in <-chan int) { // receive-only
    for v := range in { fmt.Println(v) }
}

// Fan-out: one producer, multiple consumers
func fanOut(in <-chan int, workers int) []<-chan int {
    outs := make([]<-chan int, workers)
    for i := 0; i < workers; i++ {
        ch := make(chan int)
        outs[i] = ch
        go func(out chan<- int) {
            for v := range in { out <- v }
            close(out)
        }(ch)
    }
    return outs
}

// Fan-in: merge multiple channels into one
func fanIn(channels ...<-chan int) <-chan int {
    merged := make(chan int)
    var wg sync.WaitGroup
    for _, ch := range channels {
        wg.Add(1)
        go func(c <-chan int) {
            defer wg.Done()
            for v := range c { merged <- v }
        }(ch)
    }
    go func() { wg.Wait(); close(merged) }()
    return merged
}
09 What is the difference between errgroup, WaitGroup, and context cancellation for managing goroutine lifecycles? Hard
go
// sync.WaitGroup — wait for goroutines, no error handling
var wg sync.WaitGroup
for _, url := range urls {
    wg.Add(1)
    go func(u string) {
        defer wg.Done()
        fetch(u)
    }(url)
}
wg.Wait()

// errgroup — WaitGroup + error propagation + optional cancellation
g, ctx := errgroup.WithContext(context.Background())
for _, url := range urls {
    url := url // capture loop var (pre-Go 1.22)
    g.Go(func() error {
        return fetch(ctx, url) // ctx cancelled on first error
    })
}
if err := g.Wait(); err != nil {
    log.Fatal(err)
}

// errgroup.SetLimit — bounded parallelism
g.SetLimit(10) // max 10 goroutines at once
✅ Best practice: Prefer errgroup from golang.org/x/sync/errgroup for most concurrent work. It composes context, errors, and WaitGroup into a clean API.
10 Explain Go's race detector. How does it work and what are its limitations? Medium

Go's race detector uses ThreadSanitizer (TSan) instrumentation. It tracks every memory access and synchronization event, using a happens-before algorithm to detect when two goroutines access the same memory without proper synchronization.

bash
# Enable race detector
go test -race ./...
go run -race main.go
go build -race -o myapp
go — race condition example
// ❌ DATA RACE: concurrent map write
counter := 0
var wg sync.WaitGroup
for i := 0; i < 1000; i++ {
    wg.Add(1)
    go func() {
        defer wg.Done()
        counter++ // RACE: read-modify-write not atomic
    }()
}

// ✅ Fix with atomic
var counter atomic.Int64
counter.Add(1)

Limitations:

  • 5-15x slowdown, 5-10x more memory — not suitable for production builds
  • Only detects races that actually occur at runtime — not all possible races
  • Requires sufficient test coverage to trigger the race paths

Tricky Golang Memory

Map, Slice, GC, Pointer behavior and common pitfalls

01 What causes a slice to unexpectedly share memory with its parent? How do you prevent it? Hard
go
// Slices share underlying array until one grows past capacity
original := []int{1, 2, 3, 4, 5}
sub := original[1:3]      // len=2, cap=4 (shares array!)

sub[0] = 99               // ❌ modifies original[1]!
fmt.Println(original)    // [1 99 3 4 5]

// Full slice expression limits capacity
sub2 := original[1:3:3]   // cap=2, append will allocate new array

// ✅ Explicit copy to break sharing
safe := make([]int, len(sub))
copy(safe, sub)

// ❌ Subtle bug: large slice held in memory via small sub-slice
func firstLine(data []byte) []byte {
    for i, b := range data {
        if b == '\n' {
            return data[:i] // holds ref to entire data!
        }
    }
    return data
}

// ✅ Fix: return a copy
func firstLineSafe(data []byte) []byte {
    for i, b := range data {
        if b == '\n' {
            result := make([]byte, i)
            copy(result, data[:i])
            return result
        }
    }
    return append([]byte{}, data...)
}
⚠ Memory leak pattern: Reading a 10MB file, extracting a 10-byte header as a sub-slice, then discarding the original variable — the 10MB array still lives on the heap.
02 Why is map iteration order randomized in Go? What are the implications? Easy

Go deliberately randomizes map iteration order starting from Go 1. The runtime applies a random starting bucket offset each time range is called over a map. This prevents developers from relying on undefined implementation behavior.

go
m := map[string]int{"a": 1, "b": 2, "c": 3}

// ❌ Non-deterministic — different order each run
for k, v := range m {
    fmt.Println(k, v)
}

// ✅ Sorted iteration
keys := make([]string, 0, len(m))
for k := range m { keys = append(keys, k) }
sort.Strings(keys)
for _, k := range keys {
    fmt.Println(k, m[k])
}

// ❌ Common bug: map in JSON marshaling assumes key order
// Use encoding/json — it sorts map keys alphabetically ✅
💡 Practical implication: Never write tests that depend on map ordering. Use encoding/json for deterministic serialization — it does sort keys alphabetically.
03 Explain Go's garbage collector — tricolor mark-and-sweep, write barriers, and GOGC tuning. Hard

Go uses a concurrent, tricolor mark-and-sweep GC with very short stop-the-world (STW) pauses (typically sub-millisecond). Garbage collection runs concurrently with the program.

  • White: Not yet visited (potentially garbage)
  • Gray: Visited but children not yet scanned
  • Black: Visited, all children scanned (keep)

Write barrier: Ensures no black object points directly to white — maintains the invariant during concurrent marking. Adds slight overhead to pointer writes.

bash
# GOGC=100 (default): GC when heap doubles
GOGC=100 ./app

# Higher GOGC = less frequent GC, higher memory usage
GOGC=200 ./app

# GOMEMLIMIT (Go 1.19+): hard memory ceiling
GOMEMLIMIT=256MiB ./app

# Disable GC (for benchmarks only!)
GOGC=off ./benchmark
go
// Force GC (rarely needed)
runtime.GC()

// Read GC stats
var stats runtime.MemStats
runtime.ReadMemStats(&stats)
fmt.Printf("HeapAlloc: %d MiB\n", stats.HeapAlloc/1024/1024)
fmt.Printf("NumGC: %d\n", stats.NumGC)
💡 Go 1.19+: Use GOMEMLIMIT instead of relying solely on GOGC. It prevents OOM by triggering GC more aggressively when approaching the limit.
04 What causes heap escape in Go? How do you analyze and minimize it? Hard

Escape analysis determines whether a variable can be allocated on the stack (fast, no GC) or must escape to the heap (GC-managed). Stack allocation is ~10-100x faster.

bash

// ❌ Escapes: stored in interface (boxing)

05 Explain the nil map vs empty map distinction and nil pointer dereference patterns in Go. Medium
go
var nilMap map[string]int      // nil map
emptyMap := map[string]int{} // empty map (initialized)

// Reading nil map is safe — returns zero value
v := nilMap["key"] // v = 0, no panic
_, ok := nilMap["key"] // ok = false, no panic

// ❌ Writing to nil map PANICS
nilMap["key"] = 1 // panic: assignment to entry in nil map

// Tricky: nil interface != nil pointer
type MyError struct{}
func (e *MyError) Error() string { return "err" }

func getError() error {
    var err *MyError = nil
    return err // ❌ NOT nil! interface wraps (*MyError, nil)
}

err := getError()
fmt.Println(err == nil) // false! 😱

// ✅ Fix: return nil directly
func getErrorFixed() error {
    return nil
}
⚠ Classic Go gotcha: A nil pointer wrapped in an interface is NOT equal to nil. This is the source of many subtle bugs in error handling.
06 What is sync.Pool and when should you use it? What are its limitations? Medium

sync.Pool is a goroutine-safe cache of reusable objects, reducing GC pressure by recycling allocations. Items may be removed by GC at any time — Pool does NOT guarantee persistence.

go
var bufPool = &sync.Pool{
    New: func() interface{} {
        return make([]byte, 0, 4096)
    },
}

func processRequest(data []byte) {
    buf := bufPool.Get().([]byte)
    buf = buf[:0] // reset without reallocating
    defer bufPool.Put(buf) // return to pool
    
    buf = append(buf, data...)
    // ... use buf ...
}

// Real-world: bytes.Buffer pool (used by standard library)
var bp sync.Pool
bp.New = func() interface{} { return new(bytes.Buffer) }

buf := bp.Get().(*bytes.Buffer)
buf.Reset()
defer bp.Put(buf)

Limitations:

  • Objects can be GC'd at any point — do not use for state that must persist
  • Adds complexity — only worth it when profiling shows allocation pressure
  • Pool objects are not zeroed automatically — always reset before use
07 Explain unsafe.Pointer, uintptr, and when it's valid to use unsafe in Go. Hard
go
// unsafe.Pointer: GC-tracked, can convert to any type
// uintptr: arithmetic type, NOT tracked by GC!

// ✅ Valid: reading struct field by offset (syscall layer)
type Header struct { Version uint32; Length uint32 }
h := Header{Version: 1, Length: 42}
ptr := unsafe.Pointer(&h)
lenPtr := (*uint32)(unsafe.Pointer(
    uintptr(ptr) + unsafe.Offsetof(h.Length),
))
fmt.Println(*lenPtr) // 42

// ❌ WRONG: storing uintptr across GC point
p := uintptr(unsafe.Pointer(&h))
runtime.GC() // h could be moved, p is now invalid!
_ = *(*uint32)(unsafe.Pointer(p)) // UNSAFE

// ✅ Zero-copy string↔[]byte conversion (valid in Go 1.20+)
s := "hello"
b := unsafe.Slice(unsafe.StringData(s), len(s))
⚠ Rule: Never store a uintptr across a GC-safe point. Go's GC can move objects (compaction in future versions). Always convert back to unsafe.Pointer in the same expression.

09 What is a finalizer in Go and what are the risks of using runtime.SetFinalizer? Hard

Finalizers run when the GC is about to collect an object. They're a last resort for releasing OS resources (file handles, CGO memory) when users forget to call Close().

go
type Resource struct{ fd int }

func NewResource() *Resource {
    r := &Resource{fd: openFD()}
    runtime.SetFinalizer(r, func(r *Resource) {
        closeFD(r.fd) // called by GC goroutine
    })
    return r
}

Risks and limitations:

  • Finalizer runs in a dedicated goroutine, not the creating goroutine
  • Object resurrected during finalization (finalizer has a reference to it)
  • No guaranteed order — finalizer for A may run before B even if B points to A
  • If program exits before GC runs, finalizers never run
  • Circular references prevent finalization entirely
✅ Better approach: Implement io.Closer and use defer r.Close(). Use runtime/debug.SetFinalizer only as a safety net, not primary resource management.

Git Deep Dive

Internals, workflows, rebasing, and collaboration patterns

01 Explain Git's internal object model: blobs, trees, commits, and refs. Medium

Git is fundamentally a content-addressable filesystem. Every object is stored by its SHA-1 hash under .git/objects/.

  • Blob: raw file contents (no filename)
  • Tree: directory listing (name + mode + pointer to blob/tree)
  • Commit: pointer to tree + parent(s) + author + message
  • Tag: pointer to commit with annotation
  • Refs: human-readable names pointing to commit SHAs (.git/refs/)
bash
# Inspect objects directly
git cat-file -t HEAD           # "commit"
git cat-file -p HEAD           # commit contents
git cat-file -p HEAD^{tree}    # tree contents
git rev-parse HEAD             # SHA of HEAD

# See what a ref points to
cat .git/HEAD                  # ref: refs/heads/main
cat .git/refs/heads/main       # SHA
💡 Key insight: Because objects are content-addressed, two identical files in different directories share the same blob object. Git never stores the same content twice.
02 What is the difference between git merge, git rebase, and git cherry-pick? When to use each? Medium
bash
# MERGE: creates a merge commit, preserves history
git checkout main
git merge feature/login
# History: A--B--C--M (M has two parents)
# ✅ Use for: integrating long-running branches, public branches

# REBASE: replays commits on top of target, linear history
git checkout feature/login
git rebase main
# History: A--B--C--B'--C' (new SHAs!)
# ✅ Use for: keeping feature branches up to date, clean PR history
# ❌ Never rebase shared/public branches

# Interactive rebase: squash, edit, reorder commits

# CHERRY-PICK: apply a specific commit to current branch
git cherry-pick abc1234
# ✅ Use for: hotfixes, backporting to release branches

# Cherry-pick range
git cherry-pick A..B
⚠ Golden rule: Never rebase commits that have been pushed to a shared remote branch. You will rewrite history and create divergence for all collaborators.
03 Explain git reflog and how to recover from a bad reset or accidental branch deletion. Easy

The reflog records every change to HEAD and branch refs — it's your safety net for all Git mistakes.

bash
# View reflog
git reflog

# Scenario 1: Accidental hard reset
git reset --hard HEAD~5     # Oops! Lost 5 commits
git reflog                  # Find SHA before reset
git reset --hard abc1234    # Recover to that point

# Scenario 2: Deleted branch recovery
git branch -D feature       # Oops! Branch deleted
git reflog | grep feature   # Find last commit SHA
git checkout -b feature abc1234  # Recreate branch

# Scenario 3: Find commits not reachable from any branch
git fsck --lost-found       # finds dangling commits

# Reflog expires after 90 days (default)
git config gc.reflogExpire 180.days.ago
💡 Tip: git reflog entries expire. If you need to recover very old commits, use git fsck --lost-found which scans all objects in the object store.

05 What is git bisect and how would you use it to find a regression in a Go service? Medium

git bisect performs binary search through commit history to find the first commit that introduced a bug. Reduces O(n) debugging to O(log n).

bash
git bisect start
git bisect bad                  # current HEAD is broken
git bisect good v1.2.0          # last known good tag

# Git checks out middle commit; run your test
go test ./... -run TestLatency
git bisect good                 # or bad

# Automate with a script
git bisect run bash -c '
    go build ./... 2>/dev/null &&
    go test ./internal/cache -run TestCacheEviction

💡 Pro tip: For performance regressions, use git bisect run go test -bench=BenchmarkMyFunc -benchtime=2s ./... with a script that compares benchmark output against a threshold.
06 Explain the Gitflow vs trunk-based development workflows and their tradeoffs. Medium

Gitflow: Long-lived branches (main, develop, feature/*, release/*, hotfix/*). Structured release cycle.

Trunk-based Development (TBD): All developers commit to main at least daily. Feature flags control incomplete features. Short-lived feature branches (max 1-2 days).

  • Gitflow pros: Clear release management, suitable for versioned products
  • Gitflow cons: Long-lived branches → merge hell, delayed integration
  • TBD pros: Continuous integration, fast feedback, no merge debt
  • TBD cons: Requires robust feature flags, strong test coverage, discipline
✅ Industry trend: High-performing engineering teams (Google, Meta, Netflix) use trunk-based development. Gitflow suits teams with quarterly or versioned releases (libraries, mobile apps).
07 How does git stash work internally, and what are stash pop vs stash apply differences? Easy

Stash creates two commits: one for the index (staged changes) and one for the working tree. refs/stash points to the working-tree commit, which has the index commit as a parent.

bash
# Basic stash
git stash push -m "wip: auth refactor"

# stash pop = apply + drop (removes from stash list)
git stash pop

# stash apply = apply but KEEP in stash list (safer)
git stash apply stash@{2}

# Stash including untracked files
git stash push --include-untracked

# Stash only specific files
git stash push -p  # interactive patch mode
git stash push -- go.mod go.sum

# List and inspect stashes
git stash list
git stash show -p stash@{0}

# Create branch from stash (useful for big stashes)
git stash branch feature/wip stash@{1}

02 Compare blue-green deployment, canary releases, and rolling updates. When to use each? Medium

Blue-Green: Two identical environments; traffic switched atomically via load balancer. Instant rollback by switching back.

  • ✅ Zero-downtime, instant rollback
  • ❌ 2x infrastructure cost, database migrations must be backward-compatible
  • Use for: monoliths, high-stakes releases, compliance-heavy systems

Canary Release: Route small % of traffic (1-5%) to new version, monitor metrics, gradually increase.

  • ✅ Real traffic validation, limits blast radius
  • ❌ Complex routing logic, users may see inconsistent behavior
  • Use for: major features, risky changes, high-traffic services

Rolling Update: Replace instances one-by-one (Kubernetes default). Old and new versions run simultaneously.

  • ✅ Low resource overhead, built into Kubernetes
  • ❌ Slow rollback, mixed versions during deploy
  • Use for: stateless services, backward-compatible APIs
✅ Best practice: Use canary + automated metrics gates. If error rate or latency spikes above threshold during canary, auto-rollback before wider rollout.

05 How do you handle database migrations in a CI/CD pipeline without downtime? Hard

Zero-downtime migrations require backward-compatible changes. Follow the expand-contract pattern:

  • Expand: Add new column (nullable), new table, new index. Deploy app v1 (ignores new column)

  • Contract: Deploy app v3 (only uses new column). Remove old column

        "file://migrations",
        "postgres", driver,

    if err := m.Up(); err != nil && err != migrate.ErrNoChange {

    return nil

Distributed Systems

Consensus, CAP theorem, patterns for resilient microservices

01 Explain the CAP theorem and its practical implications for designing Go microservices. Hard

CAP theorem states that a distributed system can guarantee at most two of three properties simultaneously during a network partition:

  • C (Consistency): Every read returns the most recent write
  • A (Availability): Every request receives a response
  • P (Partition tolerance): System works despite network splits

Since partitions are inevitable in distributed systems, the real choice is CP vs AP:

  • CP systems: Zookeeper, etcd, HBase — sacrifice availability for consistency. Use for: leader election, distributed locks, financial transactions
  • AP systems: Cassandra, CouchDB, DynamoDB — sacrifice consistency for availability. Use for: user sessions, shopping carts, analytics
💡 PACELC extension: Even without partitions, there's a tradeoff: Latency vs Consistency. A globally-consistent write requires consensus (slow). Eventual consistency enables local writes (fast).

02 Implement circuit breaker and retry with exponential backoff in Go. Hard
go
// Exponential backoff with jitter
func RetryWithBackoff[T any](ctx context.Context, maxAttempts int, fn func() (T, error)) (T, error) {
    var (
        zero T
        lastErr error
        base    = 100 * time.Millisecond
    )
    for attempt := 0; attempt < maxAttempts; attempt++ {
        result, err := fn()
        if err == nil { return result, nil }
        
        lastErr = err
        wait := base * time.Duration(1<<attempt)     // 100ms, 200ms, 400ms…
        jitter := time.Duration(rand.Int63n(int64(wait / 4))) // ±25% jitter
        
        select {
        case <-ctx.Done():
            return zero, ctx.Err()
        case <-time.After(wait + jitter):
        }
    }
    return zero, fmt.Errorf("all %d attempts failed: %w", maxAttempts, lastErr)
}

// Circuit breaker using sony/gobreaker
cb := gobreaker.NewCircuitBreaker(gobreaker.Settings{
    Name:        "payment-service",
    MaxRequests: 3,     // requests in half-open state
    Interval:    60 * time.Second,
    Timeout:     30 * time.Second,
    ReadyToTrip: func(counts gobreaker.Counts) bool {
        failureRatio := float64(counts.TotalFailures) / float64(counts.Requests)
        return counts.Requests >= 5 && failureRatio >= 0.6
    },
})

04 Explain the saga pattern for distributed transactions. Compare choreography vs orchestration. Hard

The Saga pattern manages distributed transactions by splitting them into local transactions with compensating actions for rollback.

Choreography: Each service publishes events; others react. No central coordinator.

  • ✅ Loose coupling, no single point of failure
  • ❌ Hard to track overall state, cyclic dependencies risk

Orchestration: A central orchestrator (saga coordinator) tells each service what to do.

  • ✅ Clear workflow visualization, easier debugging
  • ❌ Orchestrator is a bottleneck/SPF, tighter coupling
go — orchestration saga
func (s *OrderSaga) Execute(ctx context.Context, order Order) error {
    steps := []SagaStep{
        {
            Execute:    func() error { return s.inventory.Reserve(ctx, order) },
            Compensate: func() error { return s.inventory.Release(ctx, order) },
        },
        {
            Execute:    func() error { return s.payment.Charge(ctx, order) },
            Compensate: func() error { return s.payment.Refund(ctx, order) },
        },
        {
            Execute:    func() error { return s.shipping.Schedule(ctx, order) },
            Compensate: func() error { return s.shipping.Cancel(ctx, order) },
        },
    }
    
    var completed []SagaStep
    for _, step := range steps {
        if err := step.Execute(); err != nil {
            // Compensate in reverse order
            for i := len(completed) - 1; i >= 0; i-- {
                completed[i].Compensate()
            }
            return err
        }
        completed = append(completed, step)
    }
    return nil
}
05 What is consistent hashing and when would you use it in a Go distributed system? Hard

Consistent hashing maps keys to nodes on a virtual ring. When nodes are added/removed, only K/N keys remapped (vs all keys in naive modulo hashing). Virtual nodes improve load distribution.

go
type HashRing struct {
    replicas int
    ring     map[uint32]string
    sorted   []uint32
}

func (h *HashRing) Add(node string) {
    for i := 0; i < h.replicas; i++ {
        key := h.hash(fmt.Sprintf("%s:%d", node, i))
        h.ring[key] = node
        h.sorted = append(h.sorted, key)
    }
    sort.Slice(h.sorted, func(i, j int) bool { return h.sorted[i] < h.sorted[j] })
}

func (h *HashRing) Get(key string) string {
    hash := h.hash(key)
    idx := sort.Search(len(h.sorted), func(i int) bool {
        return h.sorted[i] >= hash
    })
    if idx == len(h.sorted) { idx = 0 } // wrap around
    return h.ring[h.sorted[idx]]
}

Use cases: Cache sharding (Redis Cluster), load balancer session affinity, distributed key-value stores, partitioning in message queues.

Monitoring & Observability

Metrics, tracing, logging — the three pillars

01 Explain the three pillars of observability and how they differ from monitoring. Easy

Monitoring: Tracking known failure modes with dashboards and alerts. Tells you what is broken.

Observability: Ability to understand internal system state from external outputs. Lets you answer questions you didn't know to ask. Tells you why it's broken.

  • Metrics — Aggregated numerical data over time. Low cardinality. Cheap to store. Examples: request rate, error rate, latency percentiles (RED/USE method).
  • Traces — Distributed request journey across services. High value for latency debugging. Examples: OpenTelemetry, Jaeger, Zipkin.
  • Logs — Discrete events with context. High cardinality. Expensive at scale. Examples: structured JSON logs, ELK stack, Loki.
💡 OpenTelemetry: Vendor-neutral framework for all three pillars. Write once, export to Datadog, Grafana, Jaeger, etc. This is the industry standard going forward.

04 What are SLIs, SLOs, and SLAs? How do you use error budgets? Medium
  • SLI (Service Level Indicator): Measured quantity: "99.2% of requests completed in under 200ms this week"
  • SLO (Service Level Objective): Target for an SLI: "99.9% of requests under 200ms per 30-day window"

Error budget = 1 - SLO. For 99.9% SLO = 0.1% error budget = ~43.2 min/month downtime allowed.

If error budget is exhausted:

  • Freeze new feature deployments
  • Dedicate all engineering to reliability work
  • Postmortem and root cause analysis required

06 What is the USE method vs RED method for monitoring? When to apply each? Easy

USE Method (Brendan Gregg) — For infrastructure resources (CPU, memory, disk, network):

  • Utilization — % time resource is busy
  • Saturation — work queued but not yet serviced
  • Errors — error count/rate

RED Method (Tom Wilkie) — For microservices:

  • Rate — requests per second
  • Errors — failed requests per second
  • Duration — latency distribution (p50, p95, p99)
💡 In practice: Use RED for alerting (user-visible impact), USE for diagnosis (find why RED metrics are bad). A high error rate (RED) might be explained by disk saturation (USE).

            return
        }
        w.WriteHeader(http.StatusOK)
    })
    
    // Mark ready after initialization completes
    go func() {
        warmUpCaches()
        runMigrations()
        ready.Store(true)
    }()
}
⚠ Critical distinction: Liveness failure = restart pod. Readiness failure = remove from load balancer but don't restart. Never include transient dependencies (like external APIs) in liveness checks — temporary outage would cause unnecessary pod restarts.
08 How do you write effective runbooks and postmortems for Go service incidents? Easy

Effective Runbook structure:

  • Alert description + severity + who is paged
  • Impact: what is user-visible, which SLO is affected
  • Investigation steps: specific PromQL queries, log queries, pprof commands
  • Mitigation: step-by-step rollback, feature flag disable, traffic redirect
  • Escalation path: who to call if unresolved in 30 min

Blameless Postmortem structure:

  • Timeline: what happened, when (UTC timestamps)
  • Root cause analysis (5 Whys)
  • Contributing factors (not just immediate cause)
  • What went well, what went poorly
  • Action items: specific, time-bound, owner assigned (not "improve monitoring")
✅ Key principle: Blameless postmortems. The goal is systemic improvement, not assigning fault. Systems should be designed so that individual mistakes can't cause catastrophic failures.

02 Explain cache-friendly data structures and false sharing in Go. Hard

Modern CPUs load memory in cache lines (typically 64 bytes). When two goroutines modify fields that share a cache line, they invalidate each other's cache — this is false sharing.

go
// ❌ False sharing: two counters on same cache line
type BadCounters struct {
    a atomic.Int64 // same 64-byte cache line as b
    b atomic.Int64
}

// ✅ Pad to separate cache lines
type PaddedCounter struct {
    value atomic.Int64
    _     [56]byte // 64 - 8 = 56 padding bytes
}
type GoodCounters struct {
    a PaddedCounter // owns its own cache line
    b PaddedCounter
}

// ❌ Pointer-heavy linked list — bad cache locality
// Each node traversal = potential cache miss

// ✅ Slice-based (array) structures — sequential memory access
// Prefetcher can predict and preload next elements
type Queue[T any] struct {
    items []T    // contiguous memory, cache friendly
    head, tail int
}
💡 Practical impact: False sharing can cause 10-100x performance degradation in heavily contended concurrent code. Measure with perf stat -e cache-misses on Linux or use pprof CPU profiles.

09 What is the Go memory model and how does it affect concurrent programming? Hard

The Go Memory Model defines the conditions under which reads and writes to the same variable in different goroutines are guaranteed to be observable. It uses happens-before relationships.

go
// Synchronized-before guarantees (happens-before chain)

// Channel communication — send happens before corresponding receive
done := make(chan struct{})
x := 0
go func() {
    x = 42        // write
    done <- struct{}{}  // send synchronizes
}()
<-done
fmt.Println(x) // guaranteed to see 42

// ❌ No synchronization — undefined behavior
x = 0
go func() { x = 42 }()
fmt.Println(x) // might be 0, might be 42, might corrupt

// Sync primitives that establish happens-before:
// - sync.Mutex Lock/Unlock
// - channel send/receive
// - sync.WaitGroup Done/Wait
// - sync.Once Do
// - atomic operations (with appropriate ordering)
⚠ DRF (Data Race Freedom) guarantee: Go only guarantees correct behavior for data-race-free programs. The presence of a data race makes program behavior undefined — even seemingly unrelated reads could return unexpected values.
